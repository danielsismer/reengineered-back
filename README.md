### рҹҹЎ SOLID na PrГЎtica

NГЈo como teoria de livro. Cada princГӯpio aplicado com exemplo concreto desta codebase:

#### **S вҖ” Single Responsibility Principle**
> *Cada classe tem uma Гәnica razГЈo para mudar.*

```typescript
// вқҢ Errado: Controller fazendo tudo

  async create(req, res) {
    const fruit = new Fruit(req.body); // regra de negГіcio
    await db.query('INSERT ...');       // persistГӘncia
    res.json(fruit);                    // apresentaГ§ГЈo
  }
}

// вң… Correto: Cada classe com sua responsabilidade

  constructor(private readonly useCase: CreateFruitUseCase) {}

  async handle(req: Request, res: Response): Promise<void> {
    const output = await this.useCase.execute(req.body);
    res.status(201).json(output);
  }
}
```

---

#### **O вҖ” Open/Closed Principle**
> *Aberto para extensГЈo, fechado para modificaГ§ГЈo.*

```typescript
// A interface (Port) Г© estГЎvel. Novos adaptadores nГЈo mudam o Use Case.

  findAll(): Promise<Fruit[]>;
  findById(id: FruitId): Promise<Fruit | null>;
  save(fruit: Fruit): Promise<void>;
  delete(id: FruitId): Promise<void>;
}

// Posso adicionar um RedisRepository sem tocar em CreateFruitUseCase

class FruitRepositoryPostgres implements IFruitRepository { ... }
class FruitRepositoryInMemory implements IFruitRepository { ... }
```

---

#### **L вҖ” Liskov Substitution Principle**
> *Subtipos devem ser substituГӯveis por seus tipos base.*

```typescript
// Qualquer implementaГ§ГЈo de IFruitRepository pode ser injetada
// no Use Case sem quebrar o comportamento esperado

const useCase = new CreateFruitUseCase(new FruitRepositoryPostgres(db));
// Ambos funcionam identicamente do ponto de vista do Use Case
```

---

#### **I вҖ” Interface Segregation Principle**
> *Clientes nГЈo devem depender de interfaces que nГЈo usam.*

```typescript
// вқҢ Interface gorda
interface IRepository {
  findAll(): Promise<Fruit[]>;
  findById(id: string): Promise<Fruit | null>;
  save(fruit: Fruit): Promise<void>;
  delete(id: string): Promise<void>;
  aggregate(): Promise<Stats>;      // Nem todo adapter precisa disso
  exportToCsv(): Promise<string>;   // Nem todo adapter precisa disso
}

// вң… Interfaces segregadas

interface IFruitWriter { save(fruit: Fruit): Promise<void>; }
interface IFruitDeleter { delete(id: FruitId): Promise<void>; }
interface IFruitRepository extends IFruitReader, IFruitWriter, IFruitDeleter {}
```

---

#### **D вҖ” Dependency Inversion Principle**
> *Dependa de abstraГ§Гөes, nГЈo de implementaГ§Гөes concretas.*

```typescript
// вң… Use Case depende da abstraГ§ГЈo (Port), nГЈo da implementaГ§ГЈo concreta
class CreateFruitUseCase {
  constructor(
    private readonly fruitRepository: IFruitRepository, // вҶҗ interface
  ) {}

  async execute(dto: CreateFruitDTO): Promise<CreateFruitOutput> {
    const fruit = Fruit.create(dto.name, dto.price, dto.stock);
    await this.fruitRepository.save(fruit);
    return FruitMapper.toOutput(fruit);
  }
}
```

O container de DI (InjeГ§ГЈo de DependГӘncia) cuida de conectar as abstraГ§Гөes Г s implementaГ§Гөes concretas em tempo de execuГ§ГЈo.

## вҡЎ Async-First: CRUD Robusto

Todas as operaГ§Гөes sГЈo **100% assГӯncronas**, evitando blocking I/O e garantindo throughput mГЎximo.

```typescript
// Exemplo: GetFruitsUseCase com paginaГ§ГЈo assГӯncrona
export class GetFruitsUseCase {
  constructor(private readonly repo: IFruitRepository) {}

  async execute(query: GetFruitsQuery): Promise<PaginatedOutput<FruitOutput>> {
    const [fruits, total] = await Promise.all([
      this.repo.findAll({ page: query.page, limit: query.limit }),
      this.repo.count(),
    ]);

    return {
      data: fruits.map(FruitMapper.toOutput),
      total,
      page: query.page,
      totalPages: Math.ceil(total / query.limit),
    };
  }
}
```

Uso de `Promise.all` onde operaГ§Гөes sГЈo independentes, evitando waterfalls desnecessГЎrios.
